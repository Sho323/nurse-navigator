# AI経営・加算ナビゲーションシステム (Nurse Navigator) 要件定義書 (SDD)

## 1. プロジェクト概要

*   **システム名**: AI経営・加算ナビゲーションシステム (仮称: "Nurse Navigator")
*   **目的**: 訪問看護ステーションの日々の業務（記録・情報共有）を効率化しつつ、蓄積されたデータをAIで分析することで、経営課題（入金管理、加算取得）を解決し、収益向上に貢献する統合型Webアプリケーション（PoC/MVP版）。
*   **ターゲットユーザー**: 訪問看護ステーションの経営者、管理者、事務職員、および現場の訪問看護師。
*   **アカウント構造**: 1アカウント＝1事業所。テナントIDによる完全分離。

## 2. システム構成

*   **フロントエンド**: Next.js (React), Tailwind CSS, Vercel
*   **バックエンド**: Supabase (Auth, Database, Storage, Realtime, Edge Functions)
*   **外部API**: Google Authentication, OpenAI API (GPT-4), Web Speech API
*   **デザインテーマ**: 親しみやすく温かみのある「オレンジ＆アイボリー」基調。角丸のカードUIとクリーンな余白を活かしたヘルスケアモダンデザイン。

## 3. ユーザーロールとユースケース

### 3.1 訪問看護師
*   **目的**: 現場での記録業務と情報共有を最小限の負荷で行う。
*   **ユースケース**:
    1.  スマホ（下部タブナビゲーション配置）でログインし、今日の訪問予定を確認する。
    2.  訪問先で、Web Speech APIを用いた「音声入力」でサクッと看護記録テキストを入力する。
    3.  併せて、大きく押しやすいUIでバイタル情報（体温、血圧など）や訪問種別を入力し、必要に応じて患部などの写真を添付する。
    4.  入力した内容は、即座に事業所内の「情報共有チャット（利用者別スレッド または 全体タイムライン）」へ自動共有される。

### 3.2 経営者 / 管理者
*   **目的**: 経営数値の把握と、手作業による事務負担の削減、収益機会（加算）の最大化。
*   **ユースケース**:
    1.  PC（左サイドバー配置）でログインし、ダッシュボードを開く。
    2.  日々の業務として、ダッシュボード上で売上推移グラや、現場の記録からAIが抽出した「加算取得のポテンシャル（アラート）」を確認し、現場へ指示を出す。
    3.  月末月初の事務作業として、レセプトソフトから出力した請求CSVと、金融機関から出力した入金CSVをシステムに一括アップロードする。
    4.  GPT-4を用いたAIが、明確なIDルールなしでも「よしなに」推論・突合を行った結果（消込結果）を画面で確認し、エラーや要確認の項目のみ手動で修正する。

## 4. 機能要件 (MVPスコープ)

| 機能名 | 対象ユーザー | 詳細仕様 |
| :--- | :--- | :--- |
| **Googleアカウント認証** | 全員 | Supabase Authを利用。事業所（テナント）ごとにアクセス可能なアカウントを制限。 |
| **音声・画像付き訪問記録** | 看護師、管理者 | スマホから記録入力。Web Speech APIによる音声テキスト化。写真添付（Storage保存）。バイタル情報・訪問種別の入力。 |
| **リアルタイム情報共有** | 全員 | LINE風のチャットUI。利用者ごとのスレッドと事業所全体のタイムライン。Supabase Realtimeで即時更新。 |
| **請求・入金データ取込** | 管理者のみ | レセプトCSV、銀行CSVのドラッグ＆ドロップによるアップロード。 |
| **AIによる自動消込** | 管理者のみ | GPT-4アップロードされたCSVを解析・突合。名義揺れ等もAIが推論。結果ステータス（マッチ/推論/エラー）とAIコメントを表示。 |
| **経営ダッシュボード** | 管理者のみ | 売上状況、未収金グラフの可視化。UIは大きく見やすい数字フォントを採用。 |
| **AI加算ポテンシャル分析**| 管理者のみ | 日々の看護記録テキストをバックグラウンドでGPT-4が分析。算定要件に合致する可能性をダッシュボードにアラート通知。 |

*※「AIによる画像分析」「訪問スケジュール最適化」「補助金・助成金レコメンド」は初期リリース（MVP）対象外とし、将来の拡張機能とする。*

## 5. データモデル設計（主要エンティティ）

1.  **事業所 (Tenants)**: 事業所ID、事業所名
2.  **ユーザー (Profiles)**: ユーザーID、テナントID、氏名、ロール（管理者/看護師）
3.  **利用者 (Patients)**: 利用者ID、テナントID、基本情報（氏名、カナ等）、介護・医療情報（要介護度、保険種別等）
4.  **訪問記録 (VisitRecords)**: 記録ID、テナントID、利用者ID、担当者ID、訪問日時、訪問種別、バイタル情報、記録テキスト、画像URL配列
5.  **チャットメッセージ (Messages)**: メッセージID、テナントID、送信者ID、メッセージ種別（全体/個別スレッド）、対象利用者ID（個別の場合）、本文、送信日時
6.  **売上・消込データ（SalesData）**: データID、テナントID、対象年月、請求額（CSV）、入金額（CSV）、突合ステータス、AIコメント

## 6. 非機能要件

*   **セキュリティ**: 通信の暗号化。Supabase RLSによるテナント間データ分離とロールベースアクセス制御。AIへの送信データは匿名化を推奨。
*   **UI/UX**: 医療現場の入力ストレスを軽減する、直感的でフレンドリーなモダンデザイン。レスポンシブ対応（PC・スマホ）。
*   **拡張性**: 将来的に各種システムとのAPI連携を視野に入れた汎用的な設計。
